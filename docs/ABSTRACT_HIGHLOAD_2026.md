## Тезисы (CFP — 1500–2000 знаков)

Большинство enterprise-систем строятся на реляционных БД с тысячами хранимых процедур. Понять, какая колонка откуда приходит — задача, решаемая вручную неделями. Мы автоматизировали это: парсим PL/SQL-кодовую базу, строим граф зависимостей в ArcadeDB и визуализируем его в браузере.

**Что мы сделали:**

На корпусе из 203 PL/SQL-файлов (около 70 000 строк кода) наш ANTLR4-парсер за **261 секунду** извлёк **143 000 атомов** — минимальных семантических единиц: колонки, таблицы, процедуры, DDL-операции. Граф разрешил **98.8% ссылок** — связал «колонка X из SELECT» с реальной `DaliColumn`-вершиной в YGG (нашем ArcadeDB-инстансе).

**Ключевые технические решения:**

1. **Многопроходный резолвер**: 4 прохода по AST. Pass 1 — объявления типов и таблиц; Pass 2 — рутины и их сигнатуры; Pass 3 — тела рутин; Pass 4 — `PENDING`-атомы второй попытки через пакетный LOOKUP.

2. **REMOTE_BATCH write**: вместо построчной записи в ArcadeDB — JSON-батч (`JsonlBatchBuilder`) с одним HTTP-запросом на 500 узлов. Это снижает latency записи в 40× по сравнению с наивным подходом.

3. **Геоид**: каждый атом имеет детерминированный идентификатор вида `HR.PROC_INIT.3` — уникален внутри кодовой базы, позволяет де-дупликацию при перепарсинге без полной очистки графа.

4. **Lineage за пределами SELECT**: помимо data-flow (`READS_FROM`, `WRITES_TO`), граф содержит DDL-рёбра (`DaliDDLModifiesTable/Column`), `RETURNING INTO` flow, `%ROWTYPE`-разворачивание в `DaliRecordField`.

5. **Визуализация**: React Flow + 5-уровневый drill-down (L1 схема → L2 паттерн рутин → L3 детальный граф → L4 subquery tree). Без бэкенда-агрегатора запрос к графу выполнялся бы за O(n²) — мы добавили агрегирующий GraphQL-резолвер `exploreRoutineAggregate`.

**Что рассказываем в докладе:**

- Почему ArcadeDB, а не Neo4j / Postgres (multi-model, embedded mode для preview, ArcadeQL)
- ANTLR4 grammar reuse vs ручная рекурсивная спуска
- Deadlock FRIGG-Dali при рестарте и как его устранить
- 98.8% → до 100%: какие конструкции остались (JSON_TABLE, LATERAL S9, nested records)
- Demo: live граф на реальной кодовой базе

**Целевая аудитория:** backend-разработчики, архитекторы, DBA-специалисты, работающие с legacy Oracle/PG-системами.
